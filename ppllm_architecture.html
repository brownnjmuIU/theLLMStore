Layer-by-Layer Engineering Narrative

The following descriptions detail each implemented layer of the PPLLM system — explaining what each component does, why it is technically non-trivial, and what engineering stack powers it. These narratives are intended to accompany the architecture diagram in a research paper or grant submission, demonstrating the depth and complexity of the research prototype.

Layer 0 User Interface Layer — PySide6 Desktop Application

The user-facing entry point to the PPLLM system is a native desktop application built with PySide6, Python's binding to the Qt framework. Earlier development relied on a browser-based Streamlit interface, which introduced packaging instability, localhost dependency, and port-conflict issues that made reliable end-user deployment impractical. Migrating to a native PySide6 window resolved these problems by eliminating the localhost server dependency entirely and enabling direct in-process communication between the UI and all backend pipeline components. The application presents a two-step wizard: the first step handles file selection and ingestion type routing; the second step triggers the full processing and chunking pipeline, exposes an encrypt checkbox that optionally routes output through the RSA+AES-256-GCM encryption module, and renders a real-time activity log alongside a tabbed preview of generated chunk artifacts. Distribution is handled through a GitHub Actions CI/CD pipeline that compiles the application into a cross-platform executable, enabling deployment on user machines without requiring a Python environment or manual dependency installation.

Layer 1 File Discovery & Aggregation Layer

The first data layer — built by Prabhakaran — is a file discovery and aggregation tool that systematically crawls the user's local filesystem to surface candidate files for ingestion. A FileCrawler component performs recursive directory traversal using Python's pathlib and os.walk, while a FileAnalyzer component extracts per-file metadata including MIME type, file size, last modification timestamp, and a content hash for deduplication purposes. The user is presented with a selection interface, and confirmed files are exported as a structured JSON manifest containing file paths and metadata records. Crucially, this manifest represents file paths and lightweight metadata only — not extracted content — which cleanly separates the responsibility of file discovery from the responsibility of format-specific content extraction. This separation is architecturally significant: it allows the downstream extraction layer to own all parsing logic without being contaminated by the aggregator's lightweight content previews, ensuring that the authoritative text representation of each document always comes from the dedicated extractor.

Layer 2 Permission & Access Control Layer

Before any file content is read or processed, every file access request passes through a permission and access control layer that enforces a JSON-configured rule set. A PermissionManager component loads a declarative policy configuration that governs which file types, directory paths, and content categories are permitted for ingestion under different operational modes. Each file triggers an AccessRequest evaluation against these policies before any extractor is invoked — granting or denying access based on file type, path, and metadata signals. The engineering complexity here lies not in the evaluation logic itself, but in the audit infrastructure: every access decision, whether granted or denied, is durably logged to a SQLite database (access_log.db) with a timestamp, file identifier, and policy outcome. This creates a queryable, persistent record of data provenance that is essential for privacy-preserving research: investigators and IRB reviewers can reconstruct exactly which personal files were accessed, when, under what conditions, and whether access was refused — providing the transparency guarantees that responsible personal data research requires.

Layer 3 Integration Bridge

The integration bridge (process_aggregator_manifest.py) is the orchestration script that connects the file discovery and permission layers with the core extraction and chunking pipeline. It reads the aggregator's JSON manifest, resolves each file path entry to an absolute filesystem path, optionally applies permission checks by loading the PPLLM permission module if present, and then dispatches each file to the appropriate type-specific extractor. The bridge was designed with a clear architectural principle: it deliberately ignores any content preview or summary field that the aggregator may include, relying entirely on the dedicated extractors to produce authoritative raw text. This prevents format-specific quirks — multi-column PDF layouts, PPTX slide structures, Whisper transcription decisions — from being short-circuited by lightweight metadata summaries. The bridge also carries an optional encrypt=True flag that triggers the encryption module after each chunk artifact is written, enabling the full pipeline to run in a single orchestrated pass from file discovery through to encrypted artifact storage.

Layer 4 Multi-Format Extraction Layer — Seven Specialized Extractors

The extraction layer is the most technically heterogeneous component in the system, comprising seven specialized extractors that handle every major personal file type a user might possess. PDF files are parsed with PyMuPDF, which handles multi-column layouts, embedded fonts, and document metadata with high fidelity. DOCX and PPTX files use python-docx and python-pptx respectively to extract structured text from paragraphs, tables, slide content, and speaker notes. Images are processed through pytesseract backed by the Tesseract OCR engine, converting pixel data into machine-readable text with layout-aware segmentation. Videos are handled in two stages: ffmpeg and ffprobe extract audio streams and metadata, then faster-whisper — a highly optimized implementation of OpenAI's Whisper model — performs local automatic speech recognition, running entirely on-device without any API calls or network dependency. Browser history and cookies are extracted from the SQLite databases written by Chromium-based browsers, with cookie extraction deliberately excluding value and encrypted_value fields to prevent inadvertent credential exposure. Platform data exports from services such as Google Takeout or social media archives are parsed as structured JSON records. All seven extractors produce a uniform output envelope — a raw text string and a metadata dictionary — that flows identically into the downstream processing stage, enabling the rest of the pipeline to remain format-agnostic.

Layer 5 Text Processing Layer — Clean & Chunk

After extraction, raw text passes through a two-stage text processing pipeline before it is stored or used for retrieval. The first stage is a Cleaner (cleaner.py) that normalizes whitespace, strips common extraction artifacts such as repeated newlines, control characters, Unicode directional markers, and encoding anomalies introduced by OCR and PDF parsing — producing clean, prose-ready text. The second stage is a Chunker (chunker.py) that implements a RecursiveCharacterTextSplitter strategy: it divides cleaned text into fixed-size segments with a configurable overlap window, ensuring that semantic continuity is preserved across chunk boundaries so that a sentence is never split without its surrounding context being captured in an adjacent chunk. The engineering importance of this stage lies in the metadata annotation applied to every chunk: each segment is tagged with a stable doc_id (derived from the source filename and content hash), the chunk index within the document, and the file type and path. These provenance tags are what makes the system retrieval-ready — they allow a downstream RAG layer or a conversational LLM to trace exactly which source document any retrieved passage originated from and surface that attribution transparently to the user.

Layer 6 Local Storage Layer — JSON Artifact Store

All pipeline outputs are persisted as structured JSON artifacts on the local filesystem, avoiding any external database dependency and ensuring that all personal data remains on-device throughout the core pipeline. Raw artifacts — the direct output of the extraction stage — are written to ~/Documents/LLM_Bundler/outputs/raw/, one JSON file per ingested document containing the full extracted text and metadata dictionary. Chunk artifacts — the output of the text processing stage — are written to ~/Documents/LLM_Bundler/outputs/chunks/, one JSON file per document containing an array of chunk objects each with text content, doc_id, chunk index, and source metadata. Document identity is tracked through a stable doc_id computed from a combination of the filename and a content hash, enabling safe deduplication across repeated ingestion runs of the same file without false collisions. The json_store.py helper module manages all read/write operations and enforces a consistent artifact schema across all file types, ensuring that the downstream encryption and retrieval layers can process any artifact without format negotiation.

Layer 7 Encryption Layer — RSA-2048 + AES-256-GCM Hybrid

The encryption layer provides a hybrid RSA+AES-256-GCM cryptographic scheme for securing chunk artifacts before they leave the local machine — for example, before being transmitted to a cloud-hosted LLM platform or shared with a research collaborator. The design follows industry-standard hybrid encryption: at setup time, key_manager.py generates a 2048-bit RSA keypair and persists both keys on the local filesystem at ~/Documents/LLM_Bundler/keys/. At encryption time, artifact_crypto.py generates a fresh AES-256-GCM session key for each artifact, encrypts the entire chunk JSON payload with that session key under the authenticated GCM mode (providing both confidentiality and ciphertext integrity), and then RSA-encrypts the session key using the public key. Both the RSA-wrapped session key and the GCM ciphertext are written together into a single .enc file in ~/Documents/LLM_Bundler/encrypted/. Decryption reverses this exactly: the private key unwraps the session key, the session key authenticates and decrypts the payload, and the original JSON is restored without data loss. The critical privacy property of this design is that the private key never needs to leave the originating machine — a recipient holding only the public key can receive encrypted artifacts and forward them without being able to read their contents, enabling future multi-party key exchange and federated research sharing scenarios that form the basis of the proposed future research extensions.
